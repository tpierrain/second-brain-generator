## QA harness — reproducible upgrade campaigns

The findings (especially F1) are validated by replaying a real **v3.1.0 → v3.3.0 upgrade**. To make every
pass reproducible and disposable:

- **Pristine source (immutable, read-only):** `~/legacy-brain.tgz` — a 356 MB snapshot pinned at
  **v3.1.0** (rag 1.1.0), top-level dir `legacy-brain/`, clean demo vault (7 invented notes, **no**

  source of truth for the QA starting point; treat it as append-only ground truth.
- **Disposable target:** `~/legacy-brain` (the tarball's natural extraction). Fully throwaway — recreated
  from the tgz on every pass.
- **One QA pass (the loop):**
  1. `rm -rf ~/legacy-brain` (drop the previous extraction — never touch the `.tgz`)
  2. `tar -xzf ~/legacy-brain.tgz -C ~` → fresh `~/legacy-brain` at v3.1.0
  3. open a Claude conversation **rooted in `~/legacy-brain`**, run `/update-engine`, then exercise the
     finding under test (F1 single-pass install, F2 citations, F4 restart, F5 two-modes…)
  4. if the pass mirrored the **real** Notion zone (confidential): `rm -rf ~/legacy-brain` at the end
     (F6) — never commit, never reference the zone.

> This harness keeps QA pristine-in / disposable-out, so each finding is verified against a true upgrader
> from a known-good v3.1.0 baseline, and confidential content never survives a pass.

## F1 — One update fully installs a new skill+MCP, and the brain heals itself at session start (centerpiece, ADR 0026)

**Goal:** one `/update-engine` invocation fully installs a newly-shipped skill+MCP (today it needs two),
and a brain that received code but never reconciled converges silently at the next session start.

**Two latency layers (from ADR 0026):**
- **Layer A — self-update bootstrap.** The installed `update-engine.mjs` drives the apply; Node module
  caching means a capability *introduced by* an update can only *execute* on the next run → auto-finalize
  via a fresh child process fixes this.
- **Layer B — session config-freeze.** Skills/MCP/hooks load at conversation start. A full Desktop
  restart + resumed conversation is enough (field-proven, see F4); SessionStart self-heal is the backstop.

### Files & reused building blocks (from exploration)

  - FETCH (≈145–154): `resolveLatestTag` → `fetchSource` → `readTargetManifest` (`scripts/lib/engine-fetch.mjs`)
  - CONVERGE (≈155–240): the part to extract.
- Pure libs the reconciler will reuse **unchanged** (all seam-injected, already composable):
  - `scripts/lib/engine-apply-plan.mjs` — `computeApplyPlan(target)`, `planTouches(plan, rel)` (safety core + sacred scrub)
  - `scripts/lib/mcp-reconcile.mjs` — `reconcileMcpServers({brainMcp, templateMcp, engineServerIds})`

## F7-ter — Headless probing (never boot a 2nd server) + the refocused North (ADR 0030 amended)

**Why (Thomas, 2026-06-20, long design discussion):** the F7-bis wiring (`fc2e4bb`) had the runtime probe do
a real **MCP `health_check` round-trip**. Modelling the **MCP stdio transport** precisely showed that is the
wrong shape there. The decision and its full rationale live in **[ADR 0030](../decisions/0030-engine-modules-expose-a-health-check-port-callers-own-the-policy.md)
(amended in place §3/§4 + new §6)** — this section is the execution view.

**The North (refocused):** *"never let someone work in degraded mode without knowing it."* Mission #1 = kill
**silent** degraded mode. The corrected mental model sharpens the scope: Claude already surfaces a **dead**
server (it shows `failed`, tool-calls error — the user sees it), so the **only truly silent** failure is **a
live server answering from DEGRADED DATA** (empty/stale index, missing/wrong embedder). **Catching degraded
data behind a live server is the unique value** — and it is a **headless, read-only, disk-level** job.

**The mental model (ADR 0030 §4):** a `vault-rag` MCP server is a **child process of Claude over private
stdin/stdout pipes** — no socket, so nobody can attach to the *live* server. Probing is therefore either
**(A) boot a NEW process** (a *deployment* test, spawns a 2nd server) or **(B) read disk HEADLESS**.
**Principle agreed with Thomas: a probe must NEVER boot a `vault-rag` when one may be running.**

**Division of labour:** liveness of the live server → **Claude** (not us) · data/function (index intact,
canary findable, embedder ready) → **us, headless read-only** · deployment (boots under the real config) →
**MCP smoke, once at install**.

**Probe depth (ADR 0030 §6):** **light** (file/DB reads, ms, zero ONNX) for the per-session background probe;
**full** (real embed+search) for the manual `verify-rag`. Light catches every realistic silent degraded mode;
only the rare "embedder emits bad vectors" needs full (and that is already visible).

## F8 — Obsidian, packaged as the brain's guided (optional) viewer/editor (candidate ADR 0029)

**Why (Thomas, 2026-06-20):** opening the vault in Obsidian — a proper read/write graph editor over the
**same** Markdown files Claude uses — has never been *packaged* end-to-end. Today the brain assumes Obsidian
is there and registered; in practice the user may not have it, or may not have done the one-time "Open folder
as vault". This chapter makes Obsidian a **first-class, guided, but strictly optional** capability. **Hard
guardrails (memory `obsidian-default-viewer`):** Obsidian is **recommended, never required** (notes are plain
Markdown and work without it); **no system `.md` association** (intrusive, breaks the no-lock-in promise); a
missing/unregistered Obsidian is a **soft optional nudge**, never a scary "broken" health banner.

**The zero-install path always exists:** the 🧠 local copy of a cited note opens **inline in Claude Desktop**
(a side tab) with no Obsidian at all — Obsidian is the *nicer optional* viewer/editor, not a prerequisite for
seeing the brain's own copy. This is the whole point of the citation round-trip (F2): for every answer that
refers to documents, the user can open **both** the 🔗 Notion source **and** the 🧠 local copy, with or
without Obsidian.

**Three sub-capabilities (the first two install-time, the third runtime + install post-flight):**

### F8.1 — Recommend installing Obsidian when it's absent (guided, opt-in, honest)

Claude **cannot** install Obsidian for the user: it's a downloaded desktop app (https://obsidian.md →
download → run the installer); there is **no reliable headless/brew path** (a "bare" Obsidian doesn't behave).
So the brain **guides**, it doesn't pretend to automate.

- Install-time: when the Obsidian config/app is not detected, print a clear, **opt-in** recommendation —
  *free + open-source + a viewer/editor over your **own** Markdown vault* — with the download URL and the
  one-time "Open folder as vault" step. Never block install; never nag on every run.
- Runtime: the same message is available as a **soft hint** surfaced by the F8.3 health probe (optional
  state), not the broken-capability banner.

### F8.2 — Register the brain's vault in Obsidian for the user (opt-in) — the former F3

Recurring friction: the user must manually "Open folder as vault" before `obsidian://` opens resolve.
Automate by adding an entry to Obsidian's `obsidian.json` (the F3 design, now living here).

- Config locations: macOS `~/Library/Application Support/obsidian/obsidian.json`, Windows
  `%APPDATA%\obsidian\obsidian.json`, Linux `~/.config/obsidian/obsidian.json`.
- **Pattern = `scripts/lib/open-env.mjs`** (pure builder + guard + thin I/O wrapper). Build a sibling
  `scripts/lib/obsidian-register.mjs`:
  - **Pure:** `addVaultToObsidianConfig(json, vaultPath)` — idempotent, generates the vault id, **never
    clobbers** other vaults; registers `<brain>/vault` (the path the renderer resolves).
  - **Guard:** `shouldRegisterObsidian(env, platform)` — opt-out env var (e.g. `SBG_NO_OBSIDIAN_REGISTER`),
    CI/headless off.
  - **I/O wrapper:** read → **backup** → write; detect Obsidian **installed** (no phantom config) and
    **not running** (Obsidian rewrites `obsidian.json` on quit → would clobber our edit); **fail-soft**
    to the manual instruction on any unexpected format.
- Test `scripts/lib/obsidian-register.test.mjs`: idempotence, no-clobber of existing vaults, id
  generation, fail-soft on malformed JSON, guard honored.
- Wire into `installer.mjs` as an **opt-in question** ("register your brain in Obsidian so 🧠 opens
  directly?"), slotted after RAG install / post-flight (≈ before the hand-off banner). **Install-time
  only** — the runtime reconciler must NOT touch Obsidian (ADR 0026 rejected-alternative: Obsidian may
  be running at session start).

### F8.3 — Smoke-test Obsidian as part of "is my brain healthy" (presence + registration, soft)

Extend the F7 probe registry with an **`obsidian`** probe that answers two things and reports them **softly**:

- **Installed?** — is Obsidian's config/app present on this OS (the same detection F8.2's guard uses).
- **This vault registered?** — does `obsidian.json` list `<brain>/vault` (reuse F8.2's pure reader)?
- **Status mapping (deliberate):** present + registered → `ok`; absent or not-registered → **`unknown`**
  (an OPTIONAL gap, surfaced as a soft "install/register Obsidian to open 🧠 directly" hint) — **never
  `broken`** (a working RAG doesn't need Obsidian; crying "broken" here would be a false alarm).
- **Surfacing:** the hint rides a **separate soft channel** from `formatHealthBanner`'s broken-capabilities
  banner (which stays reserved for genuine functional breakage). Reused by the **installer post-flight** so
  install-time and runtime share one Obsidian check.
- Tests: `health-probe.test.mjs` gains an `obsidian` probe case (ok / unknown — asserting it is **never**
  `broken`); the soft-hint formatter is quiet when `ok`.

### Candidate ADR 0029

"Obsidian is the recommended-but-optional vault viewer: guided install when absent + opt-in auto-register +
soft health hint; never a forced dependency, never a system `.md` file association." Carries the `Scope:`
field (Second brain runtime + Installer). Draft alongside the change.

### QA caveat (must verify by hand)

On a **brand-new** Obsidian, the very first launch shows a **vault-picker / welcome popup**; until the user
picks a vault there, `obsidian://` link opens can **stall or error** even if our `obsidian.json` edit is
correct. The install copy + the F8.3 hint must mention this one-time step, and the manual QA (below) must
exercise the first-launch path on a clean machine.
